Phase 1 — Curated structure library
Transform the viewer from a blank canvas into a browsable resource

Overview

Add a landing/browse layer to the viewer. Users arrive at a categorised gallery of landmark structures instead of an empty upload prompt. Each structure loads directly into the viewer with one click. File upload remains available for advanced users.

Data model

interface LibraryEntry {
  id: string              // e.g. "1CRN"
  accession: string       // RCSB accession code
  name: string            // common name: "Crambin"
  organism: string        // "Crambe hispanica"
  category: StructureCategory
  tags: string[]          // ["small protein","plant","disulfide bonds"]
  description: string     // 2-3 sentence plain-English summary
  atomCount: number
  resolution: number      // Å
  year: number            // year of deposition
  defaultRepresentation: RepresentationMode
  defaultColorScheme: ColorScheme
  highlightResidues?: number[]   // residues to highlight on load
}
type StructureCategory =
  | 'enzyme'
  | 'receptor'
  | 'antibody'
  | 'virus'
  | 'dna-rna'
  | 'structural'
  | 'hormone'
  | 'transport'

LibraryBrowser layout

Full-width grid above the viewer. Category pills filter the grid in real time (client-side, no fetch). Search filters by name and tag. Each StructureCard shows name, organism, category badge, atom count, and a "Load" button. Clicking load calls fetchAndLoad(accession) which fetches the mmCIF from RCSB, parses it, and dispatches to moleculeSlice. The library collapses to a slim bar once a structure is loaded.

Requirements

MUST

Library data is static (no backend). All metadata lives in library.ts. Structure files are fetched from RCSB on demand — not bundled.

MUST

Category filter and search operate client-side with no network calls.

MUST

Show a loading state on the viewer canvas while mmCIF is fetching and parsing.

MUST

Handle RCSB fetch failure gracefully: show inline error, do not crash.

SHOULD

Cache fetched mmCIF text in sessionStorage keyed by accession to avoid re-fetching on revisit within the same session.

SHOULD

Track recentlyViewed (last 5) in Zustand and persist to localStorage.

Phase 2 — Educational mode
Guided tours and learning scaffolding on top of the library

Overview

An optional guided layer that sits on top of the viewer. In educational mode the viewer becomes a step-by-step tour: the camera moves to a preset position, specific residues are highlighted, and a callout panel explains what the user is looking at. Tours are authored as static data alongside the library entries.

Data model

interface TourStep {
  id: string
  title: string
  body: string              // plain-English explanation, 2-4 sentences
  cameraPosition: [number, number, number]
  cameraTarget: [number, number, number]
  highlightResidues: number[]
  highlightChains?: string[]
  representation?: RepresentationMode   // override for this step
  colorScheme?: ColorScheme
}

interface Tour {
  id: string
  structureId: string       // links to LibraryEntry.id
  title: string
  audience: 'general' | 'undergraduate' | 'advanced'
  estimatedMinutes: number
  steps: TourStep[]
}

Tour player behaviour

Entering a tour overlays a semi-transparent panel at the bottom of the viewer. Previous/Next buttons advance steps. On each step transition: camera animates to cameraPosition/cameraTarget using a 600 ms tween (Three.js TWEEN or custom lerp), highlights update, representation switches if specified. A progress bar shows step N of total. Exit button returns to normal viewer mode and restores prior state.

Camera animation must use useTransition so the UI remains responsive during the tween. Do not block interaction.

Embed mode

Add ?embed=true&tour=haemoglobin-oxygen URL params. In embed mode: hide the library browser and controls panel, auto-start the specified tour, render in a minimal chrome suitable for iframe use. Export an embed snippet from the share panel (Phase 4).

Requirements

MUST

Tour data is static. No CMS or backend. Tours live in tours.ts alongside library data.

MUST

Camera transitions are animated (not instant). Duration 500–700 ms.

MUST

Exiting a tour restores the viewer to its pre-tour representation, colour scheme, and camera position.

MUST

Tours are only offered for the currently loaded structure.

SHOULD

Support keyboard navigation: left/right arrow keys advance tour steps.

SHOULD

Embed mode suppresses all chrome except the tour player and viewer canvas.

SHOULD

Mark completed tours in localStorage so returning users see progress.

Phase 3 — AI annotation layer
LLM-powered residue explanation and structure Q&A
This phase requires a server-side API route to hold the LLM API key. Recommended: a single Vercel Edge Function at /api/annotate. Do not expose API keys client-side.

Overview

Two AI features: (1) click any atom/residue to get a plain-English explanation of its role in the protein, and (2) a persistent Q&A panel where users can ask free-form questions about the loaded structure. Both features call the same /api/annotate edge function, which constructs a prompt from the structure metadata and the user's query.

API contract

POST /api/annotate
Content-Type: application/json

{
  "type": "residue" | "question",
  "structure": {
    "accession": string,
    "name": string,
    "organism": string,
    "category": string,
    "description": string
  },
  "residue"?: {
    "seq": number,
    "name": string,        // e.g. "HIS"
    "chainID": string,
    "secondaryStructure": string
  },
  "question"?: string
}

Response: { "answer": string }   // streaming preferred

Prompt templates

// Residue annotation
`You are a structural biology tutor. The user is viewing ${structure.name}
from ${structure.organism}, a ${structure.category}.
They have clicked residue ${residue.name} ${residue.seq} on chain
${residue.chainID} (${residue.secondaryStructure}).
Explain in 2-3 sentences what this residue does and why it matters.
Write for a curious non-specialist. No jargon without explanation.`

// Q&A
`You are a structural biology tutor. The user is viewing ${structure.name}.
${structure.description}
Answer the following question in plain English (3-5 sentences max):
${question}`

Interaction flow

Residue click (already wired in Phase 1): if AI mode is enabled, after showing SelectionInfo also fetch a residue annotation and display it in a callout below the selection info panel. Show a skeleton loader while streaming. Render streamed text token-by-token.

Q&A panel: persistent collapsible panel on the right edge. Text input at the bottom. Questions and answers stack upward in a scrollable history. Each answer includes the structure name and a timestamp. History is session-only (not persisted).

Rate limiting and cost controls

ControlImplementation
Per-session limitMax 20 annotations + 10 Q&A per session. Track in sessionStorage. Show soft warning at 80%, hard block at 100%.
Input validationEdge function rejects requests with unrecognised accession codes or questions over 500 chars.
Model choiceUse a fast, low-cost model (e.g. Claude Haiku). Residue annotations are short and latency matters.
CachingCache residue annotations in sessionStorage keyed by accession:chainID:seq. Same residue clicked twice = no second API call.

Requirements

MUST

LLM API key never leaves the server. All LLM calls go through the edge function.

MUST

AI features are opt-in. A toggle in the controls panel enables/disables the annotation layer. Default off.

MUST

Streaming responses render token-by-token. Do not wait for the full response before showing text.

MUST

All LLM errors (rate limit, timeout, 5xx) surface as user-visible inline messages, not console errors.

SHOULD

Cache residue annotations for the session to minimise API calls and latency.

MUST NOT

Make LLM calls on mouse hover. Only on explicit click or Q&A submit.

Phase 4 — Share, export and audience expansion
Get work out of the tool and broaden reach

Overview

Four distinct capabilities bundled in this phase: shareable URLs that restore viewer state, image/PNG export, embed snippet generation for course materials, and a public JavaScript API so third parties can embed and control the viewer programmatically.

1. Shareable URLs

Encode viewer state as URL search params. No backend required — all state lives in the URL.

// URL schema
?s=1HHO           // accession
&r=ribbon         // representation
&c=secondaryStructure  // colour scheme
&cx=12.4&cy=-3.1&cz=48.0  // camera position
&tx=0&ty=0&tz=0   // camera target
&hl=64,87,102     // highlighted residue seq numbers
&tour=step-3      // active tour step (optional)

On load, parse params in App.tsx before initialising the viewer. Dispatch parsed state to Zustand before first render. A "Copy link" button in the controls panel serialises current Zustand state to URL and writes to clipboard.

2. Image export

// In MoleculeViewer.ts — already scaffolded, expand to:
takeScreenshot(options?: {
  width?: number        // default: current canvas width
  height?: number       // default: current canvas height
  scale?: number        // default: 2 (retina)
  background?: string   // default: '#ffffff'
  includeLegend?: boolean
}): Promise<Blob>

Render to an offscreen canvas at the requested resolution. If includeLegend is true, composite a colour scheme legend and structure name using the Canvas 2D API before exporting. Trigger browser download via URL.createObjectURL.

3. Embed snippet

A modal in the share panel generates a ready-to-paste iframe snippet:

<iframe
  src="https://molecular-viewer-eta.vercel.app/?embed=true&s=6LU7&tour=covid-protease"
  width="800"
  height="500"
  frameborder="0"
  allowfullscreen
></iframe>

The modal lets users configure accession, starting tour, width, and height before copying. Embed mode (from Phase 2) hides all chrome except the viewer and optional tour player.

4. Public JavaScript API

Expose a minimal global API so third-party pages can control an embedded viewer via postMessage:

// Host page sends commands to the iframe
iframe.contentWindow.postMessage({
  type: 'mv:load',
  accession: '1HHO'
}, '*')

iframe.contentWindow.postMessage({
  type: 'mv:setRepresentation',
  mode: 'ribbon'
}, '*')

iframe.contentWindow.postMessage({
  type: 'mv:highlight',
  residues: [64, 87, 102]
}, '*')

// Viewer emits events back to host
window.parent.postMessage({
  type: 'mv:atomSelected',
  atom: { element: 'N', residueSeq: 87, chainID: 'A' }
}, '*')

Implement in a postMessageBridge.ts module. Document all message types in README.md under a "Public API" section.

Requirements

MUST

Shareable URLs restore accession, representation, colour scheme, and camera position exactly. Round-trip must be lossless within float tolerance (2 decimal places).

MUST

Image export works at 2× resolution minimum. Output must be a valid PNG downloadable in all modern browsers.

MUST

Embed snippet is copy-ready: one click, no editing required for default use.

MUST

postMessage handler validates message origin and type before acting. Unknown message types are silently ignored.

SHOULD

Share modal has three tabs: link, image, embed. Active tab persists in sessionStorage.

SHOULD

Image export legend includes structure name, representation mode, and colour scheme label.

MUST NOT

URL params exceed 2000 characters. If state cannot be encoded within this limit, omit highlighted residues first, then camera position.

Implement phases in order. Each phase depends on the previous: Phase 2 tours reference Phase 1 library IDs. Phase 3 AI context uses Phase 1 structure metadata. Phase 4 share URLs encode Phase 2 tour state.
